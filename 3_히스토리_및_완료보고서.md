# 📘 TOC B2B 멀티테넌시 계측 모니터링 시스템 최종 완료보고서 & 사용자 매뉴얼

본 문서는 다중 계측기 동기화 및 B2B 멀티테넌시(고객사별 독립 데이터 격리) 대시보드 시스템의 운용, 환경 설정 및 라이브 배포를 위한 종합 사용자 가이드북입니다.

## 1. 1차 개발 성과 (Google Sheets 연동 프로토타입)

초기 프로토타입은 서버나 클라우드 비용 없이 구글 스프레드시트를 클라우드 DB 대용으로 사용하는 방식으로 구축되었습니다.

1. **구글 시트 기반 물리 보안 격리 (0원 무서버 B2B)**:
   * 고객사마다 별도의 구글 스프레드시트를 개설하고 웹에 게시(.csv)하여 대시보드와 연동하였습니다.
2. **다중 기기 식별자 (`Device_ID`) 탑재**:
   * 동일한 시트 안에서 여러 대의 계측기를 가동할 수 있도록 데이터 행마다 `Device_ID`를 매립 전송하였습니다.
3. **Tkinter GUI 업로더 개발**:
   * 로컬 SQLite DB 경로, 시트 이름, 가동 주기 등을 윈도우 창에서 입력받아 구글 API로 밀어주는 인터페이스를 적용하였습니다.

*(세부 설정 방법은 구 버전 가이드라인을 참조하십시오.)*

## 2. 2차 개발 성과: Supabase 전환 및 알림 고도화 (2026년 6월 18일)

더 나은 실시간성 확보와 데이터 관리의 체계화, 그리고 관리 효율성 증대를 위해 **구글 시트 방식에서 Supabase(PostgreSQL Cloud) 데이터베이스 구조로 마이그레이션**하고, **경고 임계값 설정 및 메일 발송 자동화** 기능을 새롭게 통합하였습니다.

### A. 주요 개선 내역

1. **데이터 저장소 마이그레이션 (Supabase 전환)**:
   * 실시간성이 뛰어난 Supabase를 사용하여 데이터 적재 및 조회를 처리함으로써 속도 및 데이터 트랜잭션의 안정성을 극대화하였습니다.
2. **트렌드 차트 표시범위 최적화 (24시간)**:
   * 대시보드 접속 시 초기 데이터 표시범위를 기존 7일(`7d`)에서 **24시간(`24h`)**으로 축소하여 첫 로딩 속도를 향상시켰습니다.
3. **경고 알림 시각화 및 DB 연동**:
   * 계측 수치 상태를 **평시(흰색) · 경고(빨강)**로 구분하여 대시보드 표와 경보 램프로 시각화하였습니다. (오알람 방지를 위해 기존의 '주의' 단계는 폐기)
   * 채널별 경고 임계치와 알람 수신자 이메일 목록을 웹 대시보드에서 직접 수정할 수 있게 하였으며, 기존 DB 스키마 DDL 수정 없이 `toc_alert_high` 컬럼의 JSON 데이터 내에 포장하여 Supabase에 동기화합니다.
   * **임계치 유효성 검사 완화**: 설정 저장 시 5000 미만 임계값에 대해 뜨던 경고 확인창(confirm)을 완전히 제거하여, 50 ppm (방류수), 1000 ppm (고농도조) 등 5000 미만의 커스텀 값을 입력하더라도 경고 팝업 없이 즉시 원클릭으로 저장되도록 개선하였습니다. (0 미만의 음수가 입력되는 경우에만 정상 범위 지정을 위한 검증 메시지가 노출됩니다.)
   * 임계치 지정이 없는 경우의 기본 폴백 기준치는 다음과 같습니다:
     * **Ch 3 (방류수)**: 경고 50 (ppm)
     * **Ch 2 (고농도조)**: 경고 1000 (ppm)
     * **Ch 1 (원수조)**: 경고 2000 (ppm)
4. **로컬 SMTP 경고 메일 발송 자동화**:
   * 계측기 PC의 파이썬 업로더(`gui_uploader.py`)가 수집한 신규 데이터 중 경고 등급에 해당하는 건이 발생하면 로컬 SMTP 서버(`wsmtp.ecounterp.com`)를 경유해 수신자들에게 실시간 경보 메일을 보냅니다.
   * 메일 폭주를 막기 위해 채널별 **1시간(3600초)의 메일 발송 쿨다운 타이머**가 적용됩니다.
5. **독립 관리자 테스트 페이지 (`?admin`) 및 데이터 격리**:
   * 일반 사용자용 모니터링 화면을 방해하지 않고 알람 기능을 테스트할 수 있도록 `?admin` 주소 분기 샌드박스를 구축하였습니다.
   * **보안 통제**: 대시보드 전용 패스코드(`850`) 검증을 거친 사용자만 테스트 모드에 진입할 수 있습니다.
   * **테스트 데이터 격리**: 테스트용 모의 데이터(`Add_note` 컬럼이 `[MOCK TEST DATA]`인 행)는 메인 사용자 화면 및 차트에서 자동 제외 처리됩니다.
   * **비동기 이메일 테스트**: 웹에서 발송을 요청하면 Supabase의 `trigger_test_email: true` 플래그를 수정하고, 로컬 파이썬 uploader가 이를 감지하여 실제 발송 후 리셋하는 비동기 메커니즘을 적용했습니다.
6. **알람 인터페이스 추상화 및 팩토리 패턴 도입**:
   * 향후 카카오톡이나 MMS 알람 모듈 도입을 용이하게 하기 위해 기존 하드코딩된 이메일 발송 기능을 `alerts/` 패키지로 완전히 모듈화 및 격리하였습니다.
   * `BaseAlertSender` 추상 인터페이스를 상속받은 `EmailAlertSender` 클래스를 설계하였으며, 생성 팩토리 함수인 `get_alert_sender`를 제공합니다.
   * `uploader_config.json` 내의 `"alert_type"` 설정을 통해 메인 업로더 코드 수정 없이 알람 발송 모듈을 통째로 교체 가능한 구조를 마련하였습니다.
